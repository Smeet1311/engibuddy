# Hybrid 6-Phase PBL Framework

The Hybrid 6-Phase framework structures Project-Based Learning into six interconnected phases that build upon each other. Each phase has distinct deliverables, coaching behaviors, and exit criteria.

## Phase 1: EMPATHIZE
**Purpose:** Deeply understand the problem space, users, and constraints before proposing solutions.

**Entry Criteria:**
- Project topic assigned or chosen
- Team members identified
- Initial interest/curiosity present

**Key Activities:**
- Observe and interview end-users
- Research existing solutions
- Document user needs and pain points
- Identify stakeholder perspectives
- Define problem boundaries

**Exit Criteria:**
- Problem statement (1-2 pages) articulating "what" and "why"
- Stakeholder analysis showing who benefits/is affected
- Research summary (primary + secondary sources)
- Clearly scoped problem with defined constraints
- Team agreement on interpretation

**EngiBuddy Coaching Behavior:**
- Ask clarifying questions about users and needs
- Challenge assumptions about the problem
- Encourage primary research (user interviews, observation)
- Diagnose if scope is too broad or unclear
- Recommend "Who is your user?" and "What pain point matters most?"

**Common Stuckness Points:**
- Jumping to solutions before understanding problem
- Unclear scope (solving too broad a problem)
- Missing stakeholder analysis
- Insufficient user research
- Confusing symptoms with actual problems

---

## Phase 2: CONCEIVE
**Purpose:** Explore solution space, brainstorm alternatives, and select the most promising approach.

**Entry Criteria:**
- Clear problem statement approved
- User needs documented
- Constraints understood (time, budget, technical feasibility)

**Key Activities:**
- Brainstorm multiple solution approaches
- Evaluate feasibility of each option
- Compare technology stacks
- Prototype or sketch ideas
- Evaluate against user needs and constraints

**Exit Criteria:**
- Multiple solution concepts documented
- Selected approach justified against criteria
- Technology/methodology decisions made
- High-level architecture or design approach
- Risk assessment of chosen direction
- Team consensus on solution concept

**EngiBuddy Coaching Behavior:**
- Ask "What are other ways to solve this?"
- Challenge: "Why is this the best solution?"
- Encourage divergent thinking before convergence
- Diagnose premature convergence (only considering one solution)
- Recommend technology research or proof-of-concept exploration

**Common Stuckness Points:**
- Only considering one solution approach
- Not understanding trade-offs
- Technology choices not justified
- Overestimating feasibility
- Insufficient research on alternatives

---

## Phase 3: DESIGN
**Purpose:** Detail the solution approach with architecture, specifications, and implementation plan.

**Entry Criteria:**
- Solution concept selected and justified
- Technology choices made
- Team skills assessed

**Key Activities:**
- Create detailed architecture/design documents
- Define technical specifications
- Create Work Breakdown Structure (WBS)
- Plan milestones and timeline
- Identify resource needs
- Design testing strategy

**Exit Criteria:**
- Detailed design document with diagrams
- Complete Work Breakdown Structure
- Realistic project timeline with dependencies
- Resource/skill gap analysis
- Testing plan outline
- Quality criteria defined
- Team roles and responsibilities clear

**EngiBuddy Coaching Behavior:**
- Ask "How will you break this into smaller pieces?"
- Challenge timeline realism: "Is this achievable in your timeframe?"
- Diagnose incomplete plans (missing contingency, testing strategy)
- Encourage documentation of assumptions
- Recommend WBS review and schedule realism check

**Common Stuckness Points:**
- Unrealistic timelines
- Incomplete breakdown of work
- Missing testing/quality planning
- Vague design specifications
- Unclear team responsibilities
- No contingency planning

---

## Phase 4: IMPLEMENT
**Purpose:** Build, code, construct the solution according to design specifications.

**Entry Criteria:**
- Design approved and team understands approach
- Development environment set up
- Tools and resources available

**Key Activities:**
- Follow design specifications
- Code/build incrementally with version control
- Integrate components regularly
- Conduct code reviews
- Document decisions and changes
- Perform integration testing

**Exit Criteria:**
- Working prototype/product built
- Core features implemented
- Code properly version-controlled
- Integration completed
- Internal testing passed
- Technical documentation updated
- Blockers logged and mitigation strategies documented

**EngiBuddy Coaching Behavior:**
- Ask "How are you managing integration?"
- Diagnose technical blockers (knowledge gaps, integration issues)
- Encourage regular code review and testing
- Recommend debugging methodology
- Ask "What have you tried?" before suggesting solutions

**Common Stuckness Points:**
- Integration failures
- Technical knowledge gaps
- Scope creep (feature additions)
- Poor version control practices
- Insufficient testing during development
- No documentation of decisions

---

## Phase 5: TEST & REVISE
**Purpose:** Validate solution quality against criteria, identify and fix defects.

**Entry Criteria:**
- Working implementation exists
- Acceptance criteria defined
- Testing environment ready

**Key Activities:**
- Execute test plans
- Document defects and issues
- Prioritize and fix bugs
- Revise based on feedback
- Validate against acceptance criteria
- User testing/feedback collection
- Performance and stress testing

**Exit Criteria:**
- All critical/high-priority bugs fixed
- Acceptance criteria validated
- Test coverage documented
- User feedback incorporated
- Performance benchmarks met
- Documentation reflects final implementation
- Ready for deployment/delivery

**EngiBuddy Coaching Behavior:**
- Ask "How will you know if it works?"
- Diagnose inadequate testing (insufficient test cases, missing edge cases)
- Encourage user feedback loops
- Challenge: "Have you tested edge cases and error conditions?"
- Recommend test-driven approach to bugs

**Common Stuckness Points:**
- Inadequate test planning
- Ignoring user feedback
- Not testing edge cases
- Poor defect prioritization
- Scope creep during testing
- Missing performance validation

---

## Phase 6: OPERATE & REFLECT
**Purpose:** Deploy solution, gather feedback, and reflect on learning.

**Entry Criteria:**
- Solution meets acceptance criteria
- Documentation complete
- Team ready for deployment

**Key Activities:**
- Deploy or deliver solution
- Gather user feedback
- Document lessons learned
- Evaluate project against goals
- Create presentation/documentation
- Conduct retrospective
- Plan future improvements

**Exit Criteria:**
- Solution deployed/delivered to stakeholders
- Lessons learned documented
- Project retrospective completed
- Presentation prepared
- Portfolio artifact created
- Future improvement roadmap identified

**EngiBuddy Coaching Behavior:**
- Ask "What did you learn?" and "What would you do differently?"
- Encourage systematic reflection
- Diagnose shallow reflection (no learning documented)
- Recommend presenting findings to broader audience
- Guide portfolio artifact creation

**Common Stuckness Points:**
- Insufficient reflection
- Skipping documentation
- Not gathering/incorporating user feedback
- Poor presentations
- No portfolio artifact
- Not capturing lessons learned

---

## Cross-Phase Principles

**Iteration is Expected:** Projects don't proceed linearly. Expect to revisit phases as learning occurs.

**Documentation Builds Throughout:** Each phase deposits artifacts that inform later phases.

**Testing Starts Early:** Quality validation begins in design, not just in phase 5.

**User Stays Central:** Every phase asks "Does this solve the user's problem?"

**Constraints Are Real:** Budget, time, and team skills shape the solution.
